 Caching Strategy
The system uses Redis as a cache to make read operations faster. When a request comes in to read data (like projects or tasks), the system first checks Redis to see if the data is already stored there. If the data is found in Redis, it is returned immediately and the database is not queried. This makes the response faster and reduces database load.
If the data is not in Redis, the system fetches it from PostgreSQL using Prisma, then stores that data in Redis with a time limit (TTL), and then returns the data to the user.
When data changes (for example, a task is created, updated, deleted, a member is added or removed, or a project is created), the system deletes the related cache keys from Redis immediately. This ensures that the next read request will fetch fresh data from the database and store it again in Redis.

The TTL (time to live) is used as a backup safety mechanism. If cache invalidation fails for some reason (for example, the server crashes before deleting the cache), the cached data will automatically expire after a few minutes and be refreshed from the database.

One limitation of this approach is that if cache invalidation fails and the TTL has not expired yet, users might temporarily see outdated data. In a production system, this could be improved by using Redis pub/sub to notify all servers to clear cache, or by using a write-through caching strategy.

2. Failure Handling

The system uses BullMQ to handle export jobs in the background. If an export job fails at any step (database query, CSV creation, or file writing), the worker catches the error and updates the export record status to failed.

BullMQ is configured to retry failed jobs automatically. Each job is retried up to 3 times with exponential backoff. This means the system waits a few seconds before retrying, and the wait time increases after each failure. If the job still fails after all retries, it is moved to a failed queue.

While the job is being retried, the export status remains processing. Only after all retries fail does the status become failed permanently.

To improve monitoring in production, we could add structured logs, failure metrics, and alerts so developers can quickly detect when export jobs are failing frequently. We could also add a way for admins to see failed jobs and notify users if their export fails.

3. Access Control

The system uses Role-Based Access Control (RBAC). Authentication and authorization are handled in different layers.

Middleware is responsible only for authentication. It verifies the JWT token and attaches the user information to the request.

Controllers are kept simple. They only validate input and call the service layer.

The service layer is where authorization happens. Before reading or modifying any project or task, the service checks whether the user is a member or owner of the project. Some actions, like removing members or triggering exports, are allowed only for the project owner.

Authorization is implemented in the service layer because services contain the business logic and can be reused in different parts of the application. This also keeps controllers and middleware simple.

If a user who is not a member tries to access a project, the system returns 403 Forbidden instead of 404 Not Found. This prevents revealing whether the resource exists or not.

4. Scale

If the system grows to around 10,000 concurrent users, the first bottleneck would likely be PostgreSQL connections. Databases can only handle a limited number of connections efficiently. To solve this, we would use a connection pooler like PgBouncer, which allows many application requests to share a smaller number of database connections.

The second bottleneck would be the export worker. If many users request exports at the same time, jobs would start to queue and take longer to process. This can be solved by running multiple worker processes, since BullMQ supports multiple workers processing jobs from the same queue safely.

At larger scale, files generated by export jobs should be stored in shared storage like S3 instead of the local filesystem, so all servers and workers can access the files.

Finally, Redis itself might become a bottleneck at very large scale, so it could be moved to a Redis Cluster setup.
